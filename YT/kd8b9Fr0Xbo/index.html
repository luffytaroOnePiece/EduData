01

Mutex

Mutual Exclusion
sync.Mutex / threading.Lock
What is a Mutex?
The most fundamental synchronization primitive

A mutex (short for mutual exclusion) is a lock that ensures only one goroutine/thread can access a shared resource at a time. When one goroutine holds the mutex, all others that try to acquire it will sleep — they're blocked by the OS and consume zero CPU while waiting.

🚽 Bathroom Key Analogy: Imagine one bathroom key hanging on a hook. Only the person holding the key can use the bathroom. Everyone else queues up and waits. When they're done, they hang the key back — the next person wakes up and grabs it.
Blocking?
Yes — waiters sleep (OS blocks them)
CPU while waiting?
Zero — fully idle
Ownership?
Yes — locker must unlock it
Concurrency
1 at a time
When to use
shared counter writing to shared map updating shared struct any write-access to shared state 
// ─── Go: Mutex ───────────────────────────────────────────────────────
package main

import (
    "fmt"
    "sync"
)

var (
    counter int
    mu      sync.Mutex  // The lock — zero value is ready to use
)

func increment(wg *sync.WaitGroup) {
    defer wg.Done()

    mu.Lock()          // Goroutine grabs the lock; others SLEEP here
    defer mu.Unlock()  // Always release — even if panic occurs

    counter++          // ✅ Safe: only one goroutine runs this at a time
}

func main() {
    var wg sync.WaitGroup

    for i := 0; i < 1000; i++ {
        wg.Add(1)
        go increment(&wg)  // 1000 goroutines race to increment
    }

    wg.Wait()
    fmt.Println("Final count:", counter) // Always 1000 ✅
}

// Without mutex: counter would be anywhere from ~950 to 1000
// due to race conditions (data race = undefined behaviour)
# ─── Python: Mutex (threading.Lock) ─────────────────────────────────
import threading

counter = 0
mu = threading.Lock()  # The lock

def increment():
    global counter
    # "with mu" is idiomatic — acquires on enter, releases on exit
    with mu:                 # Same as mu.acquire() ... mu.release()
        counter += 1        # ✅ Safe: only one thread here at a time

threads = [threading.Thread(target=increment) for _ in range(1000)]
for t in threads: t.start()
for t in threads: t.join()

print(f"Final count: {counter}")  # Always 1000 ✅

# ── Note on Python's GIL ──────────────────────────────────────────────
# Python has a Global Interpreter Lock (GIL) that allows only one thread
# to execute Python bytecode at a time. This means counter++ MIGHT work
# without a mutex for CPython, but it's NOT guaranteed — always use locks
# for correctness. The GIL is being removed in Python 3.13+ (free-threaded).
Don't unlock from a different goroutine/thread than the one that locked — mutex ownership belongs to the locker. Unlocking twice causes a panic in Go.
In Go, defer mu.Unlock() immediately after mu.Lock() is the standard pattern. It guarantees the unlock runs even if the function panics.
02

Counting Semaphore

Concurrency Limiter
channel / threading.Semaphore
What is a Counting Semaphore?
Controls how many can work simultaneously

A semaphore holds an internal count N. Acquiring it decrements the count; releasing increments it. When the count hits zero, new acquirers block. Unlike a mutex (N=1), a semaphore can allow multiple goroutines in the critical section simultaneously — up to N.

🅿️ Parking Lot Analogy: A parking lot has 3 spaces. Up to 3 cars (goroutines) can be inside simultaneously. The 4th car must wait at the barrier until someone leaves. The semaphore count = available spaces.
Blocking?
Yes — when count = 0
CPU while waiting?
Zero — goroutine sleeps
Ownership?
No — anyone can release
Concurrency
Up to N at a time
When to use
limit DB connections rate-limit HTTP calls worker pool concurrency cap throttle expensive operations 
// ─── Go: Counting Semaphore via buffered channel ─────────────────────
package main

import (
    "fmt"
    "sync"
    "time"
)

// Buffered channel with capacity N = semaphore with N slots.
// Sending = acquire (take a slot). Receiving = release (free a slot).
const maxConcurrent = 3
var sem = make(chan struct{}, maxConcurrent) // 🅿️ 3 parking spaces

func worker(id int, wg *sync.WaitGroup) {
    defer wg.Done()

    sem <- struct{}{}        // Acquire: "I'm entering the lot"
    defer func() { <-sem }() // Release: "I'm leaving" (always runs)

    // At most 3 workers are between these lines simultaneously
    fmt.Printf("Worker %d started\n", id)
    time.Sleep(200 * time.Millisecond) // Simulate real work
    fmt.Printf("Worker %d done\n", id)
}

func main() {
    var wg sync.WaitGroup

    for i := 1; i <= 10; i++ { // 10 workers, only 3 run at once
        wg.Add(1)
        go worker(i, &wg)
    }
    wg.Wait()
    fmt.Println("All done")
}

// Tip: golang.org/x/sync/semaphore provides a weighted semaphore
// useful when some tasks are "heavier" and should count as 2 or 3 slots.
# ─── Python: Counting Semaphore ──────────────────────────────────────
import threading
import time

# threading.Semaphore(N) — built-in counting semaphore
sem = threading.Semaphore(3)    # Only 3 threads at a time

def worker(thread_id):
    with sem:                     # acquire on enter, release on exit
        print(f"Worker {thread_id} running")
        time.sleep(0.2)
        print(f"Worker {thread_id} done")

threads = [
    threading.Thread(target=worker, args=(i,))
    for i in range(1, 11)       # 10 threads, only 3 concurrent
]
for t in threads: t.start()
for t in threads: t.join()

# ── asyncio version (non-blocking I/O) ───────────────────────────────
# import asyncio
# sem = asyncio.Semaphore(3)
# async def worker(i):
#     async with sem:
#         await asyncio.sleep(0.2)  # non-blocking wait
In Go, a buffered channel is the idiomatic semaphore — no library needed. struct{}{} uses zero memory as the token.
03

Spin Lock

Busy-Wait Lock
atomic.CompareAndSwap / threading
What is a Spin Lock?
Keeps checking in a tight loop instead of sleeping

A spin lock does NOT put the goroutine to sleep. Instead it loops continuously ("spins") checking if the lock is free. This burns CPU but avoids the overhead of an OS context switch (~1–10 µs). Only beneficial when the lock is held for very short durations.

🔄 Impatient Person Analogy: Instead of sitting in the waiting room (sleeping), you stand right outside the door and keep knocking every millisecond. Wasteful if the wait is long, but faster than signing in at reception if someone's just washing their hands.
Blocking?
No — active loop
CPU while waiting?
High — burns cycles
Ownership?
Yes
Best for
Sub-microsecond holds
When to use
kernel code / OS internals interrupt handlers lock held < 1µs real-time / low-latency systems 
// ─── Go: Spin Lock via atomic ─────────────────────────────────────────
package main

import (
    "fmt"
    "runtime"
    "sync"
    "sync/atomic"
)

// SpinLock uses a single int32: 0 = unlocked, 1 = locked
type SpinLock struct {
    state int32
}

func (s *SpinLock) Lock() {
    for {
        // CompareAndSwap atomically: if state==0 then state=1 (locked), return true
        // If state==1 already (someone else holds it), CAS returns false → keep spinning
        if atomic.CompareAndSwapInt32(&s.state, 0, 1) {
            return // We grabbed the lock!
        }
        // Yield to the Go scheduler — prevents complete CPU starvation of other goroutines
        runtime.Gosched()
    }
}

func (s *SpinLock) Unlock() {
    atomic.StoreInt32(&s.state, 0) // Atomic write: state = 0 (unlocked)
}

var (
    sl      SpinLock
    counter int
)

func main() {
    var wg sync.WaitGroup
    for i := 0; i < 1000; i++ {
        wg.Add(1)
        go func() {
            defer wg.Done()
            sl.Lock()
            counter++  // Protected section
            sl.Unlock()
        }()
    }
    wg.Wait()
    fmt.Println("Counter:", counter) // 1000
}

// ⚡ Spin vs Mutex — when does spin win?
//   Spin wins when: hold time < context-switch overhead (~1-5µs)
//   Mutex wins when: hold time is long or many goroutines compete
# ─── Python: Spin Lock via atomic flag ───────────────────────────────
import threading
import time

# Python doesn't have atomic CAS like Go, but we can simulate
# a spin lock using a simple flag + busy-wait
class SpinLock:
    def __init__(self):
        self._lock = threading.Lock()  # Underlying OS lock
        self._spinning = False

    def acquire(self):
        while True:
            # try_lock: non-blocking acquire attempt
            if self._lock.acquire(blocking=False):
                return                # Got it!
            # Couldn't get it — spin (yield CPU slice)
            time.sleep(0)            # Cooperative yield to other threads

    def release(self):
        self._lock.release()

# Usage
sl = SpinLock()
counter = 0

def increment():
    global counter
    sl.acquire()
    try:
        counter += 1
    finally:
        sl.release()     # Always release, even on exception

threads = [threading.Thread(target=increment) for _ in range(500)]
for t in threads: t.start()
for t in threads: t.join()
print(f"Counter: {counter}")

# ⚠️  In Python, spin locks rarely make sense due to the GIL.
# Use threading.Lock() directly for production code.
Never use a spin lock when the lock is held for more than a few microseconds — you'll burn entire CPU cores doing nothing useful. In user-space Go code, sync.Mutex is almost always the right choice.
04

Binary Semaphore

Signaling Primitive
channel(1) / threading.Semaphore(1)
What is a Binary Semaphore?
A semaphore with max count = 1, but no ownership

A binary semaphore looks like a mutex — only 0 or 1 — but the key difference is ownership. A mutex must be unlocked by the same goroutine that locked it. A binary semaphore can be signaled by goroutine A and waited on by goroutine B. This makes it ideal for event notification.

🚦 Traffic Light Analogy: One person (producer) controls the green light; another (consumer) waits at the red. The person at the light doesn't care who turned it green — they just go when it is. No "ownership" of the signal.
Count range
0 or 1 only
Ownership?
No — key difference from mutex
Primary use
Event signaling
Cross-goroutine?
Yes — A signals, B waits
When to use
producer → consumer notification task completion signal start gate (release N goroutines) one-shot event 
// ─── Go: Binary Semaphore via channel ────────────────────────────────
package main

import (
    "fmt"
    "time"
)

// A channel of capacity 1 IS a binary semaphore.
// send = "signal"  |  receive = "wait"
var ready = make(chan struct{}, 1)

func producer() {
    time.Sleep(100 * time.Millisecond)
    fmt.Println("Producer: data is ready!")
    ready <- struct{}{}  // Signal: "consumer, you can go now" 🟢
}

func consumer() {
    fmt.Println("Consumer: waiting for data...")
    <-ready              // Wait: block until producer signals
    fmt.Println("Consumer: got signal, processing!")
}

func main() {
    go producer()
    consumer()  // main goroutine waits here
}

// ── Start-gate pattern: release ALL goroutines simultaneously ─────────
func startGateExample() {
    gate := make(chan struct{}) // Unbuffered — blocks all waiters

    for i := 0; i < 5; i++ {
        go func(id int) {
            <-gate  // All 5 goroutines park here
            fmt.Printf("Goroutine %d off!\n", id)
        }(i)
    }

    time.Sleep(10 * time.Millisecond) // Let all goroutines reach the gate
    fmt.Println("3... 2... 1...")
    close(gate)  // close() broadcasts to ALL receivers simultaneously 🚀
                 // Unlike send (wakes 1), close wakes everyone at once
}
# ─── Python: Binary Semaphore ─────────────────────────────────────────
import threading
import time

# threading.Semaphore(1) — binary semaphore
# threading.BoundedSemaphore(1) — same, but raises error if released too many times
signal = threading.Semaphore(0)    # Start at 0 — consumer will block immediately

def producer():
    time.sleep(0.1)
    print("Producer: data ready!")
    signal.release()                # Signal: increment count from 0 → 1

def consumer():
    print("Consumer: waiting...")
    signal.acquire()                # Wait: blocks until count > 0, then decrements
    print("Consumer: processing!")

t = threading.Thread(target=producer)
t.start()
consumer()
t.join()

# ── Mutex vs Binary Semaphore — the key difference ───────────────────
#
#   Mutex:
#     thread A: lock()   ← acquires ownership
#     thread A: unlock() ← MUST be the same thread
#
#   Binary Semaphore:
#     thread A: release()  ← signals (no ownership)
#     thread B: acquire()  ← waits and consumes the signal
#
# Use binary semaphore for signaling between DIFFERENT threads.
# Use mutex for protecting data accessed by the SAME thread context.
In Go, closing an unbuffered channel is the idiomatic "broadcast" — it wakes all goroutines waiting on <-ch at once. Sending only wakes one.

Read-Write Mutex

Read-Optimized Lock
sync.RWMutex / threading.RLock
What is a Read-Write Mutex?
Concurrent reads, exclusive writes

Reading shared data from multiple goroutines simultaneously is safe — they don't modify anything. A regular mutex would block all readers unnecessarily. An RWMutex allows unlimited concurrent readers, but when a writer needs access, it gets exclusive access — all readers and other writers are blocked until it's done.

📚 Library Analogy: Many people can read books from the shelf simultaneously (shared read lock). But when the librarian needs to reorganize shelves (write lock), she closes the library — no one reads or writes until she's done. Readers never block each other.
Multiple readers?
Yes — all hold RLock simultaneously
Multiple writers?
No — exclusive
Reader+Writer?
No — writer blocks readers
Best pattern
reads >> writes
When to use
in-memory cache config store lookup tables read-heavy shared maps service registry 
// ─── Go: RWMutex — concurrent reads, exclusive writes ────────────────
package main

import (
    "fmt"
    "sync"
)

type SafeCache struct {
    mu    sync.RWMutex        // RWMutex instead of Mutex
    store map[string]string
}

func (c *SafeCache) Get(key string) (string, bool) {
    c.mu.RLock()          // READ lock: many goroutines can hold this at once ✅
    defer c.mu.RUnlock()  // Release read lock
    v, ok := c.store[key]
    return v, ok
}

func (c *SafeCache) Set(key, val string) {
    c.mu.Lock()           // WRITE lock: exclusive — blocks all readers AND writers ⛔
    defer c.mu.Unlock()   // Release write lock
    c.store[key] = val
}

func main() {
    cache := &SafeCache{store: make(map[string]string)}
    cache.Set("name", "Alice")

    var wg sync.WaitGroup

    // 20 concurrent readers — all run simultaneously (no blocking between them)
    for i := 0; i < 20; i++ {
        wg.Add(1)
        go func() {
            defer wg.Done()
            v, _ := cache.Get("name") // 20 goroutines read simultaneously ✅
            _ = v
        }()
    }

    // 1 writer — waits for all current readers to finish, then gets exclusive access
    wg.Add(1)
    go func() {
        defer wg.Done()
        cache.Set("name", "Bob") // Writer blocks until all readers done ⛔→✅
    }()

    wg.Wait()
    v, _ := cache.Get("name")
    fmt.Println("name:", v)
}

// Performance: RWMutex can be 5-10x faster than Mutex for read-heavy workloads
# ─── Python: Read-Write Lock ──────────────────────────────────────────
import threading

# Python's threading module doesn't ship an RWLock, so we build one.
# Pattern: track reader count; writers wait for zero readers.
class RWLock:
    def __init__(self):
        self._readers = 0
        self._read_lock  = threading.Lock()     # Protects _readers count
        self._write_lock = threading.Lock()     # Exclusive for writers

    def acquire_read(self):
        with self._read_lock:
            self._readers += 1
            if self._readers == 1:    # First reader blocks writers
                self._write_lock.acquire()

    def release_read(self):
        with self._read_lock:
            self._readers -= 1
            if self._readers == 0:    # Last reader unblocks writers
                self._write_lock.release()

    def acquire_write(self):
        self._write_lock.acquire()    # Exclusive — blocks all readers+writers

    def release_write(self):
        self._write_lock.release()

# Usage
rw = RWLock()
cache = {}

def read_cache(key):
    rw.acquire_read()
    try:
        return cache.get(key)       # Multiple threads can be here at once
    finally:
        rw.release_read()

def write_cache(key, val):
    rw.acquire_write()
    try:
        cache[key] = val            # Exclusive access
    finally:
        rw.release_write()

# Tip: The 'rwlock' pip package provides a production-ready version.
# pip install rwlock
# from rwlock import RWLock
Don't upgrade an RLock to a Lock in the same goroutine — deadlock! Release the read lock first, then acquire the write lock.
If writes are frequent (>30% of operations), a plain Mutex may outperform RWMutex due to lower overhead. Profile first.

Priority Inheritance Mutex

Advanced / RT Systems
custom / POSIX PTHREAD_PRIO_INHERIT
What is Priority Inversion?
When a low-priority task starves a high-priority one

In systems with task priorities, priority inversion occurs when: a low-priority task holds a lock, a high-priority task needs it and blocks, but a medium-priority task (needing no lock) runs freely — effectively making the high-priority task wait behind medium. Priority inheritance solves this by temporarily boosting the low-priority task's priority to match the waiting high-priority task.

👑 Classic Example — Mars Pathfinder (1997): NASA's Mars rover experienced resets due to priority inversion. A low-priority task held a shared bus lock, a high-priority task blocked on it, but a medium-priority task kept running. The watchdog timer saw the high-priority task not completing and reset the system. Priority inheritance was patched remotely.

       blocked (waiting for lock)              blocks briefly, then runs

MED    ░░░░████████████░░░░            MED    ░░░░░░░░░░░░████░░░░
       runs freely (no lock needed)            waits for HIGH to finish

// ─── Go: Simulating Priority-Aware Mutex ─────────────────────────────
package main

import (
    "context"
    "fmt"
    "sort"
    "sync"
    "time"
)

type PriorityRequest struct {
    name     string
    priority int       // higher = more urgent
    ready    chan struct{}
}

type PriorityMutex struct {
    mu       sync.Mutex
    queue    []*PriorityRequest   // waiters sorted by priority
    queueMu  sync.Mutex
    locked   bool
}

func (pm *PriorityMutex) Lock(ctx context.Context, name string, priority int) error {
    req := &PriorityRequest{
        name:     name,
        priority: priority,
        ready:    make(chan struct{}),
    }

    pm.queueMu.Lock()
    pm.queue = append(pm.queue, req)
    // Keep queue sorted: highest priority first
    sort.Slice(pm.queue, func(i, j int) bool {
        return pm.queue[i].priority > pm.queue[j].priority
    })
    pm.queueMu.Unlock()

    // Try to acquire in a goroutine — this queues us
    go func() {
        pm.mu.Lock()       // Block here until the actual mutex is free
        close(req.ready)  // Signal: we have the mutex now
    }()

    select {
    case <-req.ready:
        fmt.Printf("[%s] priority=%d: acquired lock ✅\n", name, priority)
        return nil
    case <-ctx.Done():
        fmt.Printf("[%s] priority=%d: timed out ⏰\n", name, priority)
        return ctx.Err()
    }
}

func (pm *PriorityMutex) Unlock(name string) {
    fmt.Printf("[%s] released lock 🔓\n", name)
    pm.mu.Unlock() // Frees the underlying mutex — highest-priority waiter will get it next
}

func main() {
    pm := &PriorityMutex{}

    // Low-priority task grabs the lock first, holds it for 300ms
    go func() {
        ctx := context.Background()
        pm.Lock(ctx, "low-task", 1)
        fmt.Println("low-task: doing slow work...")
        time.Sleep(300 * time.Millisecond)
        pm.Unlock("low-task")
    }()

    time.Sleep(10 * time.Millisecond) // Ensure low grabs lock first

    // High-priority task needs the lock — gets it BEFORE any medium-priority waiters
    ctx, cancel := context.WithTimeout(context.Background(), 500*time.Millisecond)
    defer cancel()

    if err := pm.Lock(ctx, "high-task", 10); err != nil {
        fmt.Println("Priority inversion: high-task could not get lock!")
    } else {
        time.Sleep(50 * time.Millisecond)
        pm.Unlock("high-task")
    }

    time.Sleep(500 * time.Millisecond) // Let everything finish
}

// 📌 For true priority inheritance: use os/exec + POSIX threads (cgo),
//    or design with priority queues + golang.org/x/sync/semaphore
# ─── Python: Priority-Aware Lock with heapq ──────────────────────────
import threading
import heapq
import time

class PriorityLock:
    """
    Lock that serves waiters in priority order.
    Higher priority number = served first.
    """
    def __init__(self):
        self._lock     = threading.Lock()         # Underlying mutex
        self._heap     = []                       # Min-heap of (-priority, event)
        self._heap_mu  = threading.Lock()         # Protect the heap

    def acquire(self, priority: int, name: str, timeout=None):
        event = threading.Event()

        with self._heap_mu:
            # negate priority so heapq (min-heap) serves highest first
            heapq.heappush(self._heap, (-priority, name, event))

        # Spin-wait until we're at the front of the queue AND we can get the lock
        deadline = time.time() + (timeout if timeout else float('inf'))
        while time.time() < deadline:
            with self._heap_mu:
                top = self._heap[0][2]        # Event of highest-priority waiter

            if top is event and self._lock.acquire(blocking=False):
                with self._heap_mu:
                    heapq.heappop(self._heap)
                print(f"[{name}] priority={priority}: acquired ✅")
                return True
            time.sleep(0.005)                # Yield — check again soon

        # Timeout: remove from queue
        with self._heap_mu:
            self._heap = [(p,n,e) for p,n,e in self._heap if e is not event]
            heapq.heapify(self._heap)
        print(f"[{name}] priority={priority}: timed out ⏰")
        return False

    def release(self, name: str):
        print(f"[{name}] released 🔓")
        self._lock.release()

# Usage
pl = PriorityLock()

def task(name, priority, hold_time, delay):
    time.sleep(delay)
    if pl.acquire(priority, name, timeout=1.0):
        time.sleep(hold_time)
        pl.release(name)

threads = [
    threading.Thread(target=task, args=("low",  1,  0.3, 0.00)),
    threading.Thread(target=task, args=("high", 10, 0.1, 0.01)),
    threading.Thread(target=task, args=("med",  5,  0.1, 0.01)),
]
for t in threads: t.start()
for t in threads: t.join()
Go and Python do not expose OS-level thread priority control. For true POSIX priority inheritance (PTHREAD_PRIO_INHERIT), you need C/cgo or a real-time OS. The patterns above are user-space approximations.

Side-by-Side Comparison

Quick Reference

Mutex 1 at a time Yes Zero (sleeps) Protect shared state sync.Mutex threading.Lock
Counting Semaphore Up to N No Zero (sleeps) Limit concurrency chan (buf N) threading.Semaphore
Spin Lock 1 at a time Yes High (spins) Ultra-short critical sections atomic.CAS Lock(blocking=False)

Decision Guide — Which should I use?
Need to protect shared data from concurrent writes? → Mutex
Mostly reading, rarely writing? → RWMutex
Limit how many goroutines run a section at once? → Counting Semaphore
Notify another goroutine that something happened? → Binary Semaphore / channel
Lock held for nanoseconds, ultra-hot path? → Spin Lock (use carefully)
Real-time system, need to prevent priority inversion? → Priority Inheritance Mutex
Communicating data between goroutines? → Channel (Go's first choice)
Go's mantra: "Do not communicate by sharing memory; instead, share memory by communicating." Reach for channels first, mutexes when you genuinely need shared state.
